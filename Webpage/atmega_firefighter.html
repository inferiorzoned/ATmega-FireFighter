Introduction

As a part of CSE 316: Microcontrollers, Microprocessors and Embedded Systems sessional course, we have created a prototype of an automated and primary fire prevention system as term project. We named the project ATmega FireFighter.

On 20 February 2019, a massive fire accident took place at Chowkbazar, Dhaka. At least 80 people were dead and 50 others were injured in that accident[1]. Also a tragic video of that accident became viral on social media. We were deeply touched by that event. A series of fire accidents also took place around that time. So we wanted to build a prototype which can take an instant primary action against fire accident happened in home or office place.

We also thought, as this project includes sensors, motors, GSM module and also we may need to trigger some actions in case of a particular event e.g. fire, we can learn a significant amount about the practical use of microcontroller.

A Brief Description

ATmega FireFighter is an ATmega32-based project. In this project, we designed a system that can take an instant primary action to extinguish or at least stop spreading of the fire. The system also sends an alert SMS to the home owner or authority's mobile number when it detects fire incident. The project consists of a hardware part and a software part. In the hardware part, we had to setup some devices and establish a circuit connection. Details are given in the Hardware Part section. In the software part, we had to develop two programs in C language to control the relevant ATmega32 microcontrollers. Details are given in the Software Part section.

How It Works?

A flame(IR) sensor detects the existence of fire. The flame sensor and the opening of the water pipe are set parallely directed at the same direction. This subsystem is placed over a servo motor. The servo motor constantly rotates from about -75° to 75° and thus moves the sensor and helps to check fire. When the sensor gets any fire signal, it halts and check for confirmation about fire existence. As soon as, it is confirmed about a fire incident, the GSM module sends an alert SMS to the preset mobile number, e.g. authority's number. Then it starts throwing water at the fire incident as long as the sensor detects the existence of fire. If the fire is extinguished, the servo motor again starts rotating and checking for fire.

Algorithm

This is a high-level algorithm of our system's operation:

  1. initialize all register
  2. start sweeping servo motor
  3. read data from flame sensor
  4. perform analog-to-digital conversion on that data
  5. store converted data into sensor_reading
  6. if sensor_reading < FIRE_THRESHOLD
    1. stop servo motor
    2. read more values from sensor
    3. if sensor_reading is persistent i.e. fire is confirmed
      1. send SMS from GSM module
      2. turn on pump to throw water and move the servo motor at small angle keeping the fire event at the center
      3. to check if fire is extinguished goto step(3)// if sensor_reading < FIRE_THRESHOLD
    4. else continue sweeping the servo motor
  7. else continue sweeping the servo motor

Hardware Part

Our hardware part consists of a flame sensor, a servo motor, a motor driver, a dc motor, a gsm module and a lcd module. In this section, we will briefly show the design of our total hardware systems as well as the working principles of the mentioned components.

Flame Sensor

The flame sensor is actually an IR sensor. It can detect the IR radiation caused from fire events. The sensor we used has a digital output pin. But while working with it, we discovered this pin gives analog output too. Actually the pin gives an output voltage ranging from 0 to 5V. So we converted this analog data to digital data using the ADC(analog-to-digital converter) of ATmega32. After ADC operation we were able to represent the sensor reading as a value ranging from 0 to 1023. Values near 1023 mean 'No fire' and lower values mean the sensor has detected IR emission.

Servo Motor

Servo motor is an electromechanical actuator which is very useful to move some object to a particular position by rotation and to hold that there. We used servo motor to rotate the flame sensor and the water nozzle. To control the servo motor, we had to feed a control signal from the microcontroller to the servo. We used the PWM feature of ATmega32 to generate the control signal.

To generate PWM signal, we used Timer1 module of ATmega32 microcontroller which is a 16 bit timer. Timer1 has two PWM channels(A and B). We used ATmega32's internal clock frequency which is 1MHz. We took 1 as prescaler to get the original clock frequency as timer frequency. So, the timer got 1μs time period. We also used the Fast PWM mode. In our selected timer mode, ICR1 gets the top value of the timer.

fpwm = fcpu⁄N(1+TOP)
=> TOP = fcpu⁄N × fpwm - 1
=> TOP = 1 MHz⁄1 × 50 Hz - 1 = 19999
So, we set ICR1 = 19999

According to the datasheet, to set the servo to the center position we need to feed the servo a PWM signal of 1.5 ms uptime. To set the servo to it's leftmost position the signal uptime should be 1 ms and to set it to it's rightmost position the signal uptime should be 2 ms. However, in practical we had to tweak the values a little to get our expected result. The values we used are given in the table below.

Position Pulse Width(μs)
Full left 900
Center 1500
Full right 2100
Table 2: Servo Motor position with corresponding pulse width

The main problem of the servo motor is that it cannot give a feedback according to the current situation. We can give it command to move to a certain position, but we cannot stop it before it reaches that position. So we had to be tricky to solve this problem. Instead of sweeping the whole area by a single movement i.e. from left to all the way right, or vice versa, we moved the servo by smaller steps. In this way, we can check the sensor output after every step and can take the decision whether we should halt the servo or not. Still there was error when the decision had to be taken between consecutive steps, but the error margin was small and overall this idea was a good workaround.

Motor Driver and Pump Motor

We used a 12V DC motor to pump out and throw water to the fire event. To control the DC motor we used a motor driver. A motor driver can control upto two motors. So, it has two pairs of input and output pins. As we need to control a single motor we used the pair-1 pins(EN1, IN1, IN2, OUT1, OUT2). Motor status as well as rotational direction can be controlled using four types of combination of control signal to pin IN1 and IN2. The combinations are shown in the table below. GSM Module

GSM stands for Global System for Mobile Communication which is a standard to describe the protocols for second-generation (2G) digital cellular networks used by mobile devices. GSM module gives us the facility to establish a communication between a mobile-device and the module. We used A6 GSM module for our project. A6 GSM module needs an USB cable to power up. The RXD pin was connected with ATmega32's TXD(pin 15) and the TXD pin was connected with ATmega32's RXD(pin 14) to transmit data between the GSM module and the microcontroller. At first we established a network between the GSM module and the ATmega32. Then whenever our sensor detected a fire occurance, we sent a message to the respective authority about the fire incident from the GSM module's sim card using ATmega32's UART communication.
The detailed wire connections of the GSM module with the ATmega32 microcontroller is showed in the Detailed Pin Diagram.

LCD Module

We used a 16x2 LCD module to show the value read from the flame sensor and show relevant messages according to the situation. We know values near 1023 mean 'No fire'. During a 'No fire' situation, we only printed the value from the flame sensor. When the IR reading from the sensor went below our threshold value which most possibly meant a fire incident had occured (the fire incident was ensured after observing some repetitive values), we printed a message showing "Fire!!!" and also printed the corresponding sensor_reading in the next line. After the pump had been started by the motor driver, we used the LCD module to show the message "Pump ON" and in the next line we showed the message "Throwing Water" until the fire had been extinguished by the water pump. In the LCD module, we used a potentiometer to control the contrast of the LCD module.The wire connections of the LCD module with the ATmega32 microcontroller is showed in the Detailed Pin Diagram.

Software Part

The code for ATmega FireFighter is written from scratch, except the lcd.h header file. lcd.h is built by electroSome. The code is written in c language and compiled with the compiler of Atmel Studio IDE. We also used ExtremeBurner to load the hex file into the ATmega32 microcontroller.

The software was designed in a way such that the program maintains some state according to the actual state of the system. The program takes necessary actions when a change of state is observed.

At initialization state, all corresponding registers are initialized with appropriate values. Input and output pins are defined. We also set the servo motor to center position. When all the initialization is complete, the program enters sweeping state.

In the sweeping state, we incrementally change the value of OCR1A to move servo to the right until it reaches it's rightmost position. Then the direction of movement is reversed. Thus the system checks the area infront of it. While the servo is moving, the program reads analog data from the flame sensor, performs an ADC operation and shows the value on lcd module. Whenever sensor data gives value under a certain threshold, program enters fire detection state.

In the fire detection state, the program halts the servo and check if the values provided by the flame sensor are consistent. If the values are inconsistent the program assumes that as an false alarm and goes back to the sweeping state. Otherwise it enters extinguishing state.

In this state, the program is confirmed about the fire incident. So at first it sends the secondary microcontroller(used to control the GSM) a signal. After recieving the signal, the microcontroller sends an alert message to a preset mobile number using the GSM module. Meanwhile the primary microcontroller turns on the pump motor by sending signal to the motor driver. In this state, the program moves the servo back and forth in a smaller angle by writing values to OCR1A, first incrementally and then decrementally. This movement keeps the fire event at the center and also materialize the idea of stopping the spreading of fire. After a time period, the program again enters the fire detection state to check whether the fire is extinguished.

As we used a seperate microcontroller just to control the GSM module, we needed to write a seperate program for that microcontroller. Using two ATmega gives us the ability to perform two crutial tasks(extinguishing fire and sending alert message) parallely. We used the UART feature of ATmega32 to control the GSM module.

As this software runs in a embedded system with limited memory and processing power, we tried to make our code as efficient as we could. Besides we wanted our code to be readable as well as understandable. The codes are also well documented.

Problem Faced

As the flame sensor is sensible to the IR emission, presense of the sunlight was a real trouble. However we were able to reduce the problem by calibrating the sensor. Also the sensor should have a field of view of 60° angle according to the manufacturer. But in practice it has a field of view of approximately 10° angle. So the sensor can only detect the fire event if it happens at the same plane directly in front of it.

The servo should move from -90° to 90° according to the datasheet. But in reality, the servo moves from approximately -75° to 75°. This narrows the surveillance area of the system. Also to achieve this much coverage we had to find a better working value of the uptime of PWM signal by trial and error.

Using a GSM module with an atmega32 microcontroller might create lots of troubles in a hardware project. The frequency of the atmega32 was not similar to that of the GSM module. So, it created a huge problem to sync the frequencies of them. We solved it in the code. We had to fix a baud rate for the serial communication established between the atmega32 microcontroller and the GSM module. For this, we set specific values in the USART registers. The signals recieved by the GSM could not be transmitted smoothly most of the time, the reason of which was that the GSM was not configured correctly at first. Also we needed to power it directly from another 5V source to make it work fine.

Due to inconsistent power supply sometimes, the Lcd module misses some bits during passing data from the microcontroller to the module and hence shows garbage values. We also faced similar issues. In such case, we had to reconnect the module's power supply to fix it back to work smoothly again.

Sometimes due to the unexpected behaviour of the flame sensor, the system detected fire although the fire incident was happening at a small angular distance from the direction the sensor was pointing at. To solve this problem, we had to increase the range of water-throwing. We also made a sprinkler and got better result at extinguishing fire.

Conclusion

This firefighter works for a short range. Also, the flame sensor cannot move in the z-axis. So it is understandable that this prototype is not fit to work in a three dimensional plane.

Nevertheless, this is a very low-cost solution to prevent any fire hazard inside any residence. Our future target with this project is to make the sensor movable in the z-axis. Also, we have a plan to make a movable bot out of it so that it can move,detect fire at any place inside the walls and extinguish it.
